Output: Fail-fast logic integrated into the relist pipeline and critical notifications.
</objective>

<task type="auto">
  <name>Task 2: Implement Fail-Fast in RelistEngine</name>
  <files>logic/relist_engine.py</files>
  <action>
    1. Define a custom exception `ConsoleSessionError(Exception)` at the top of the file.
    2. In `RelistEngine.process_cycle`, immediately after navigation (`_navigate_with_retry`), add a check:
       - If `self.auth.is_console_session_active(self.page)` or `self.bot_state.is_console_locked()` is True:
         - Log a critical error: "Console session detected - aborting relist to prevent BAN."
         - Raise `ConsoleSessionError("Console session active")`.
    3. Ensure this check happens BEFORE the Golden Sync and BEFORE the scan.
  </action>
  <verify>
    <automated>grep -E "ConsoleSessionError|is_console_session_active" logic/relist_engine.py</automated>
  </verify>
  <done>RelistEngine aborts the cycle immediately upon console session detection.</done>
</task>

<task type="auto">
  <name>Task 3: Add Final Guards in RelistExecutor</name>
  <files>browser/relist.py</files>
  <action>
    1. Modify `RelistExecutor.relist_single` and `RelistExecutor.relist_all`.
    2. At the very beginning of both methods, add a guard check:
       - If `self.auth.is_console_session_active(self.page)` is True:
         - Log "CRITICAL: Console session detected just before click. Aborting action."
         - For `relist_single`, return a `RelistResult` with `success=False` and `error="Console session active"`.
         - For `relist_all`, return a `RelistBatchResult` with a `relist_error="Console session active"`.
    3. This ensures that even if a session is detected mid-cycle, no click is performed.
  </action>
  <verify>
    <automated>grep -E "is_console_session_active" browser/relist.py</automated>
  </verify>
  <done>RelistExecutor prevents any clicks if a console session is detected at the moment of execution.</done>
</task>
